# US-E1-03 · Battery Real-Time Telemetry (D-02)

## User Story

**As a** developer building the PoC data layer, **I want to** provide a live operational snapshot of each BESS asset with current telemetry readings, **so that** the Fleet Overview table, Current State card, Portfolio Header KPIs, and the F-06 engine can consume real-time operational data.

## Description

Live operational snapshot of each asset. Stored in `battery_telemetry.json` as an array of one record per asset. Each record contains: id (FK to D-01), timestamp, soc_pct, soh_pct, power_kw (positive = charging, negative = discharging), mode (charging/discharging/idle/fault), temperature_c, voltage_v, current_a, next_action (Charge/Discharge/Hold), next_action_window, fault_code (null unless fault).

## Acceptance Criteria

- **AC-1**: The dataset contains exactly 12 telemetry records.
- **AC-2**: Each record contains all required properties: id, timestamp, soc_pct, soh_pct, power_kw, mode, temperature_c, voltage_v, current_a, next_action, next_action_window, fault_code.
- **AC-3**: Each id matches a valid asset ID from D-01.
- **AC-4**: mode is one of: "charging", "discharging", "idle", "fault".
- **AC-5**: Mode/power sign consistency: charging → power_kw > 0, discharging → power_kw < 0, idle/fault → power_kw ≈ 0.
- **AC-6**: soc_pct and soh_pct are between 0 and 100 inclusive.
- **AC-7**: fault_code is non-null only when mode = "fault".
- **AC-8**: Fault-mode assets have next_action = "Hold" and next_action_window = "—".
- **AC-9**: Timestamps use UTC-offset format (Europe/Amsterdam).
