# US-E1-04 · Battery Historical Time Series (D-03a / D-03b)

## User Story

**As a** developer building the PoC data layer, **I want to** provide per-asset time series of power and state of charge at 15-minute resolution in two files (24h for Replay Card, 7d for AI Engine), **so that** F-05 can render the replay chart and F-06 can compute DoD baselines and calendar aging.

## Description

Two files with the same schema serve two consumers:
- `battery_history_24h.json` (D-03a): Last 24 hours, 96 intervals per asset — consumed by F-05 Replay Card.
- `battery_history_7d.json` (D-03b): Last 7 days, 672 intervals per asset — consumed by F-06 engine Steps 2c and 3c.

Both are objects keyed by asset ID, each value an array of intervals in ascending timestamp order. Each interval has: timestamp, power_kw, soc_pct.

Derived fields: Total charged = Σ max(power_kw, 0) × 0.25; Total discharged = Σ max(−power_kw, 0) × 0.25; Net cycles = (charged + discharged) ÷ 2 ÷ capacity_kwh; Average DoD over 7d (from D-03b); Time above 80% SoC in prior 24h (from D-03a).

Consistency: D-03a last soc_pct = D-02 soc_pct per asset. D-03b last soc_pct = D-03a first soc_pct (shared boundary).

## Acceptance Criteria

- **AC-1**: D-03a contains entries for all 12 assets, each with exactly 96 intervals.
- **AC-2**: D-03b contains entries for all 12 assets, each with exactly 672 intervals.
- **AC-3**: Each interval contains timestamp, power_kw, and soc_pct.
- **AC-4**: Timestamps are spaced exactly 15 minutes apart in ascending order.
- **AC-5**: soc_pct is between 0 and 100 at every interval.
- **AC-6**: D-03a last interval soc_pct equals D-02 soc_pct for each asset (SoC continuity).
- **AC-7**: D-03b last interval soc_pct equals D-03a first interval soc_pct (shared boundary).
- **AC-8**: SoC direction is consistent with power sign.
- **AC-9**: No gaps, duplicate timestamps, or out-of-order entries.
