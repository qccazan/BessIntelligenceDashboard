# US-E1-09 · Cross-Dataset Consistency Rules

## User Story

**As a** developer building the PoC data layer, **I want to** ensure all mock datasets are mutually consistent according to defined cross-referencing rules, **so that** the dashboard never displays contradictory values between cards, tables, or KPI chips.

## Description

These cross-dataset constraints must hold in the generated mock data or the dashboard will show contradictory values. They govern relationships between D-01 through D-07.

## Acceptance Criteria

- **AC-1**: SoC continuity — D-03a last interval soc_pct equals D-02 soc_pct for every asset. D-03b last interval soc_pct equals D-03a first interval soc_pct.
- **AC-2**: Mode / power sign — D-02 mode = charging ⇒ power_kw > 0; discharging ⇒ power_kw < 0; idle/fault ⇒ power_kw ≈ 0.
- **AC-3**: Fault assets held — Any asset with D-02 mode = "fault" must have D-05 per_battery_actions action = "Hold".
- **AC-4**: Recommendation aligns with prices — D-05 charge_window corresponds to lowest-price hours in D-04; discharge_window to highest.
- **AC-5**: Portfolio KPIs — D-02 power_kw summed across all assets = Net Power chip; Σ(soc_pct × capacity_kwh / 100) = Available Now chip.
- **AC-6**: Per-battery action timing — D-02 next_action and next_action_window match D-05 per_battery_actions for every asset.
- **AC-7**: D-06 / D-04 time alignment — D-06 hour_start values exactly match D-04 prices[].hour_start for the same 24-hour horizon.
- **AC-8**: D-06 night-time irradiance — solar_irradiance_wm2 = 0 for all D-06 records where hour_start falls between 20:00 and 06:00 local time.
