# US-00-08 · Cross-Dataset Consistency Rules

## User Story

**As a** developer building the PoC data layer, **I want to** ensure all six mock datasets are mutually consistent according to defined cross-referencing rules, **so that** the dashboard never displays contradictory values between cards, tables, or KPI chips.

## Description

These cross-dataset constraints must hold in the generated mock data or the dashboard will show contradictory values. The rules govern relationships between D-01 (master data), D-02 (telemetry), D-03 (historical time series), D-04 (market prices), and D-05 (AI recommendation). They ensure SoC continuity, mode/power sign consistency, fault handling, recommendation alignment with prices, portfolio KPI accuracy, and per-battery action timing.

## Acceptance Criteria

- **AC-1**: SoC continuity — D-03 last interval `soc_pct` equals D-02 `soc_pct` for every asset.
- **AC-2**: Mode / power sign — D-02 mode = "charging" implies `power_kw` > 0; "discharging" implies `power_kw` < 0; "idle" or "fault" implies `power_kw` ≈ 0.
- **AC-3**: Fault assets held — Any asset with D-02 mode = "fault" has D-05 `per_battery_actions` action = "Hold".
- **AC-4**: Recommendation aligns with prices — D-05 `charge_window` corresponds to the lowest-price hour(s) in D-04; `discharge_window` corresponds to the highest.
- **AC-5**: Portfolio KPIs — D-02 `power_kw` summed across all assets equals the Net Power chip value; sum of (`soc_pct` × `capacity_kwh` / 100) equals the Available Now chip value.
- **AC-6**: Per-battery action timing — D-02 `next_action` and `next_action_window` match the corresponding entry in D-05 `per_battery_actions` for every asset.
